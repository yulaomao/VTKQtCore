# VTKQtCore 软件编写指南

这份文档是面向 **第一次接触这个框架的人** 写的。

目标只有一个：**让你看完后，可以基于当前项目独立做出一个完整软件，能加模块、能切模块、能连 Redis、能让 logic 控 UI，也能让 UI 控 logic。**

如果你时间有限，建议先按下面顺序阅读：

1. 先看“整体架构”
2. 再看“创建一个新软件”
3. 再看“创建一个新模块”
4. 再看“UI / logic / Redis / Node 交互”
5. 最后照着“从 0 到 1 实战模板”做一遍

---

## 1. 先用一句话理解这个项目

这个项目本质上是一个 **Qt + VTK + Redis 的模块化医疗软件框架**。

它把系统拆成了几层：

- **软件装配层**：决定当前软件有哪些模块、模块顺序是什么、初始模块是什么
- **UI 层**：页面、侧边栏、底部栏、3D 窗口，以及页面内部的动作构建
- **协调层**：负责把 UI 和 logic 接起来
- **logic 层**：处理业务逻辑、节点操作、模块间调用、Redis 访问
- **场景层**：统一管理 PointNode / LineNode / ModelNode / TransformNode
- **通信层**：Redis 轮询、Redis 订阅、控制消息路由

你可以把它理解成：

> `软件 = 主窗口壳 + 多个模块 + 一个统一的 LogicRuntime + 一个统一的 SceneGraph + 一个统一的 CommunicationHub`

## 2. 你写软件时最重要的几个核心类

先记住下面几个类，后面所有内容都围着它们转：

### 2.1 软件装配入口

- `src/main.cpp`
- `src/app/software/BaseSoftwareInitializer.h`
- `src/app/software/ConcreteSoftwareInitializers/DefaultSoftwareInitializer.cpp`

作用：

- 读取软件类型
- 创建 `LogicRuntime`
- 创建 `MainWindow`
- 创建具体 `SoftwareInitializer`
- 注册模块 logic
- 注册模块 UI
- 注册通信源
- 进入初始模块

### 2.2 模块 UI 协调器

- `src/ui/coordination/ModuleCoordinator.h`
- `src/ui/coordination/ApplicationCoordinator.h`
- `src/ui/coordination/UiActionDispatcher.h`

作用：

- `ModuleCoordinator`：负责一个模块的主页面、附属 UI、通知分发，以及给页面注入模块级 `UiActionDispatcher`
- `ApplicationCoordinator`：负责整个软件的模块切换、壳层通知、右侧/底部挂载，以及持有壳层级 `UiActionDispatcher`
- `UiActionDispatcher`：负责把页面或壳层构建好的 `UiAction` / `command` 发给 `ILogicGateway`，也负责切模块和 resync 请求

### 2.3 模块逻辑基类

- `src/logic/registry/ModuleLogicHandler.h`

作用：

- 每个模块都要有一个自己的 `ModuleLogicHandler`
- 它处理 UI 发来的动作
- 它处理 Redis 推过来的状态
- 它能操作 `SceneGraph`
- 它能直接读写 Redis
- 它能调用其他模块的 logic

### 2.4 逻辑总调度器

- `src/logic/runtime/LogicRuntime.h`

作用：

- 管所有模块 logic
- 接收 UI Action
- 路由到对应模块
- 处理模块切换
- 接收轮询 / 订阅 / 控制消息
- 统一发出 `LogicNotification`

### 2.5 主窗口壳

- `src/shell/WorkspaceShell.h`

主窗口区域分为：

- `CenterStack`：中间主页面区域
- `RightShellHost`：右侧壳层区域
- `RightAuxiliaryHost`：右侧当前模块附属区域
- `BottomShellHost`：底部壳层区域
- `BottomAuxiliaryHost`：底部当前模块附属区域

这决定了“模块多个 UI 如何放到指定位置”。

## 4. 整体运行流程，先建立全局脑图

下面这张“文字脑图”非常重要：

### 4.1 软件启动流程

1. `main.cpp` 创建 `LogicRuntime`
2. 创建 `RedisGateway` / `CommunicationHub`
3. 从 Redis 读 `software_profile` 和 `current_software_type`
4. `SoftwareInitializerFactory::create()` 创建具体软件初始化器
5. `initializer->initialize(...)`
6. 初始化器内部做这些事：
   - 建 `PageManager`
   - 建 `GlobalUiManager`
   - 建 `ApplicationCoordinator`
   - 注册模块 logic
   - 注册模块 UI
   - 注册壳层模块
   - 注册 Redis 通信源
   - 发送初始模块切换动作
7. `LogicRuntime` 切到初始模块
8. `ApplicationCoordinator` 把对应模块 UI 挂到主窗口里

### 4.2 UI 控 logic 流程

`按钮点击 -> Page 自己构建 command / payload -> UiActionDispatcher -> ILogicGateway -> LogicRuntime -> 对应 ModuleLogicHandler::handleAction`

补一句当前版本最重要的变化：

> **页面自己按规则构建消息，再交给注入进来的 `UiActionDispatcher` 发送；`ModuleUiAssemblers` 不再负责把每个按钮 signal 翻译成 action。**

### 4.3 logic 控 UI 流程

`ModuleLogicHandler emit logicNotification -> LogicRuntime -> ILogicGateway -> ApplicationCoordinator / ModuleCoordinator -> Page / Panel 更新`

### 4.4 Redis 进 logic 流程

#### 轮询

`PollingTask -> CommunicationHub -> LogicRuntime::onStateSampleReceived -> handleStateSample`

#### 订阅

`SubscriptionSource -> CommunicationHub -> LogicRuntime::onStateSampleReceived -> handleStateSample`

#### 控制消息

`CommunicationHub::controlMessageReceived -> LogicRuntime::onControlMessageReceived -> 转成 UiAction -> handleAction`

## 5. 如何创建一个新软件

用户问的“创建新软件”，在这个项目里不是“新建一个 EXE 工程”，而是：

> **在同一个框架里，定义一种新的软件装配方案。**

比如：

- 软件 A：启用 `params + planning + navigation`
- 软件 B：启用 `datagen + pointpick`
- 软件 C：模块一样，但初始模块、顺序、通信配置不一样

### 5.1 你需要改哪些地方

最少改 3 处：

1. 新建一个 `XXXSoftwareInitializer.h/.cpp`
2. 在里面自注册到 `SoftwareInitializerFactory`
3. 让 Redis 里的 `current_software_type` 或 `software_profile.softwareType` 指向它

### 5.2 最简单参考实现

参考：

- `src/app/software/ConcreteSoftwareInitializers/PlanningNavigationSoftwareInitializer.cpp`
- `src/app/software/ConcreteSoftwareInitializers/DefaultSoftwareInitializer.cpp`

`PlanningNavigationSoftwareInitializer` 就是一个很标准的“新软件”示例。

它做的事很简单：

- 继承 `DefaultSoftwareInitializer`
- 覆盖启用模块列表
- 覆盖模块显示顺序
- 覆盖初始模块

### 5.3 新软件模板

```cpp
// src/app/software/ConcreteSoftwareInitializers/MySoftwareInitializer.h
#pragma once

#include "DefaultSoftwareInitializer.h"

class MySoftwareInitializer : public DefaultSoftwareInitializer
{
    Q_OBJECT

public:
    explicit MySoftwareInitializer(const QString& softwareType,
                                   RunMode mode,
                                   QObject* parent = nullptr);

    QStringList getEnabledModules() const override;
    QStringList getModuleDisplayOrder() const override;
    QString getInitialModule() const override;
};
```

```cpp
// src/app/software/ConcreteSoftwareInitializers/MySoftwareInitializer.cpp
#include "MySoftwareInitializer.h"
#include "SoftwareInitializerFactory.h"

namespace {

const bool s_registeredMySoftware = [] {
    SoftwareInitializerFactory::registerInitializer(
        QStringLiteral("my_software"),
        [](const QString& softwareType, RunMode mode, QObject* parent) -> BaseSoftwareInitializer* {
            return new MySoftwareInitializer(softwareType, mode, parent);
        });
    return true;
}();

}

MySoftwareInitializer::MySoftwareInitializer(const QString& softwareType,
                                             RunMode mode,
                                             QObject* parent)
    : DefaultSoftwareInitializer(softwareType, mode, parent)
{
}

QStringList MySoftwareInitializer::getEnabledModules() const
{
    return {
        QStringLiteral("params"),
        QStringLiteral("planning"),
        QStringLiteral("navigation")
    };
}

QStringList MySoftwareInitializer::getModuleDisplayOrder() const
{
    return {
        QStringLiteral("params"),
        QStringLiteral("planning"),
        QStringLiteral("navigation")
    };
}

QString MySoftwareInitializer::getInitialModule() const
{
    return QStringLiteral("params");
}
```

### 5.4 如果你还想自定义通信配置

可以覆盖：

- `registerCommunicationSources(CommunicationHub* commHub)`
- `registerShellModules(...)`
- `configureAdditionalSettings(...)`

参考文件：

- `src/app/software/ConcreteSoftwareInitializers/DefaultSoftwareInitializer.cpp`

### 5.5 软件类型是怎么被选中的

在 `main.cpp` 里：

```cpp
RedisSoftwareResolver resolver(...);
const QVariantMap softwareProfile = resolver.resolveSoftwareProfile();
QString softwareType = softwareProfile.value("softwareType").toString();
if (softwareType.isEmpty()) {
    softwareType = resolver.resolveSoftwareType();
}

BaseSoftwareInitializer* initializer =
    SoftwareInitializerFactory::create(softwareType, runMode, &app);
```

也就是说，你有两种方式指定软件：

#### 方式 1：Redis key

- `current_software_type = my_software`

#### 方式 2：Redis JSON 配置

- `software_profile.softwareType = my_software`

### 5.6 小白最容易犯的错

#### 错误 1：只写了类，但没注册工厂

现象：

- Redis 明明设置成新软件类型，但启动还是老软件

原因：

- 没有 `SoftwareInitializerFactory::registerInitializer(...)`

#### 错误 2：启用了模块，但没注册 UI 或 logic

现象：

- 切模块报错
- 页面空白
- `No module handler registered`

原因：

- `registerModuleLogicHandlers()` 或 `registerModuleUIs()` 漏写

#### 错误 3：顺序里写了没启用的模块

虽然 `BaseSoftwareInitializer` 会尽量帮你过滤，但你仍然应该保证：

- `getModuleDisplayOrder()` 里的模块，都在 `getEnabledModules()` 里

## 6. 如何创建一个新模块

这一节最重要。

在这个项目里，一个标准模块通常包含：

1. 一个 `ModuleLogicHandler`
2. 一个主页面 `QWidget`
3. 可选的一个或多个附属 UI（右侧 / 底部）
4. 一个模块命令定义头，例如 `ReviewUiCommands.h`（推荐）
5. 一个 UI 装配函数，放到 `ModuleUiAssemblers.cpp`
6. 在某个 `SoftwareInitializer` 里注册

### 6.1 推荐目录结构

建议放到：

`src/modules/你的模块名/`

比如新模块叫 `review`，建议是：

```text
src/modules/review/
  ReviewModuleLogicHandler.h
  ReviewModuleLogicHandler.cpp
    ReviewUiCommands.h
  ReviewPage.h
  ReviewPage.cpp
  ReviewPage.ui
  ReviewStatusPanel.h
  ReviewStatusPanel.cpp
  ReviewStatusPanel.ui
  review.qrc            // 如果你有模块自己的图片资源
```

### 6.2 第一步：写 logic

参考最简单示例：

- `src/modules/pointpick/PointPickModuleLogicHandler.cpp`
- `src/modules/params/ParamsModuleLogicHandler.cpp`

模板：

```cpp
// ReviewModuleLogicHandler.h
#pragma once

#include "logic/registry/ModuleLogicHandler.h"

class ReviewModuleLogicHandler : public ModuleLogicHandler
{
    Q_OBJECT

public:
    explicit ReviewModuleLogicHandler(QObject* parent = nullptr);

    void handleAction(const UiAction& action) override;
    void handleStateSample(const StateSample& sample) override;
    void onModuleActivated() override;
    void onModuleDeactivated() override;
    void onResync() override;
};
```

```cpp
// ReviewModuleLogicHandler.cpp
#include "ReviewModuleLogicHandler.h"
#include "ReviewUiCommands.h"

ReviewModuleLogicHandler::ReviewModuleLogicHandler(QObject* parent)
    : ModuleLogicHandler(QStringLiteral("review"), parent)
{
}

void ReviewModuleLogicHandler::handleAction(const UiAction& action)
{
    if (action.actionType != UiAction::CustomAction) {
        return;
    }

    const QString command = action.payload.value(QStringLiteral("command")).toString().trimmed();

    if (command == ReviewUiCommands::approve()) {
        // 处理 UI 发来的 approve 动作
        return;
    }
}

void ReviewModuleLogicHandler::handleStateSample(const StateSample& sample)
{
    Q_UNUSED(sample);
    // 处理 Redis 轮询或订阅数据
}

void ReviewModuleLogicHandler::onModuleActivated()
{
    // 模块切进来时执行
}

void ReviewModuleLogicHandler::onModuleDeactivated()
{
    // 模块切走时执行
}

void ReviewModuleLogicHandler::onResync()
{
    // 通信重同步时执行
}
```

### 6.3 第二步：写主页面 UI

参考：

- `src/modules/pointpick/PointPickPage.h`
- `src/modules/pointpick/PointPickPage.cpp`

页面职责要尽量简单：

- 接收用户点击
- 按模块规则构建 `command` / `payload`
- 调用注入进来的 `UiActionDispatcher` 发送动作
- 接收 logic 通知后刷新控件

不要在页面里写业务逻辑。

模板：

```cpp
class ReviewPage : public QWidget
{
    Q_OBJECT

public:
    explicit ReviewPage(QWidget* parent = nullptr);
    void setActionDispatcher(UiActionDispatcher* dispatcher);

public slots:
    void setStatusText(const QString& text);
    void setApproved(bool approved);

private:
    UiActionDispatcher* m_actionDispatcher = nullptr;
};
```

一个最小可用的页面点击写法大致是：

```cpp
#include "ReviewUiCommands.h"
#include "ui/coordination/UiActionDispatcher.h"

connect(m_approveButton, &QPushButton::clicked, this, [this]() {
    if (!m_actionDispatcher) {
        return;
    }

    m_actionDispatcher->sendCommand(ReviewUiCommands::approve());
});
```

也就是说，**新模块默认写法是“页面自己发 action”，而不是“页面发 signal，装配层再翻译”。**

### 6.4 第三步：写附属 UI（可选）

如果你希望模块右侧有状态面板，或底部有工具条，就再写额外 QWidget。

参考：

- `src/modules/pointpick/PointPickStatusPanel.cpp`

它就是一个很好的“右侧附属面板”例子。

### 6.5 第四步：在 `ModuleUiAssemblers.cpp` 里装配 UI

参考：

- `src/app/software/ModuleUiAssemblers.cpp`

这里是所有模块 UI 的装配中心。

一个最标准的新模块 UI 装配函数长这样：

```cpp
void registerReviewModuleUi(const ModuleUiAssemblyContext& context)
{
    auto* coordinator = new ModuleCoordinator(
        QStringLiteral("review"),
        context.gateway,
        context.applicationCoordinator);

    auto* page = new ReviewPage();
    page->setActionDispatcher(coordinator->getActionDispatcher());
    auto* statusPanel = new ReviewStatusPanel(context.mainWindow->getWorkspaceShell());

    coordinator->setMainPage(page);
    coordinator->addAuxiliaryWidget(
        statusPanel,
        ModuleCoordinator::AuxiliaryRegion::Right);

    context.pageManager->registerPage(QStringLiteral("review"), page);
    context.applicationCoordinator->registerModuleCoordinator(coordinator);

    QObject::connect(coordinator, &ModuleCoordinator::notificationForPage,
                     page, [page, statusPanel](const LogicNotification& notification) {
                         if (notification.eventType != LogicNotification::CustomEvent) {
                             return;
                         }

                         page->setStatusText(notification.payload.value(QStringLiteral("status")).toString());
                         statusPanel->setStatusText(notification.payload.value(QStringLiteral("status")).toString());
                     });
}
```

这里要特别记住：

> `ModuleUiAssemblers.cpp` 现在主要做的是 **创建页面 / 创建 coordinator / 注入 dispatcher / 连接 notification**，不再应该堆一长串“某个按钮 signal -> 某个 action”的翻译代码。

### 6.6 第五步：声明装配函数

别忘了在：

- `src/app/software/ModuleUiAssemblers.h`

里加声明：

```cpp
void registerReviewModuleUi(const ModuleUiAssemblyContext& context);
```

### 6.7 第六步：在软件初始化器里注册 logic 和 UI

比如在 `DefaultSoftwareInitializer.cpp`：

```cpp
void DefaultSoftwareInitializer::registerModuleLogicHandlers(LogicRuntime* runtime)
{
    if (isModuleEnabled(QStringLiteral("review"))) {
        runtime->registerModuleHandler(new ReviewModuleLogicHandler(runtime));
    }
}

void DefaultSoftwareInitializer::registerModuleUIs(MainWindow* mainWindow,
                                                   LogicRuntime* runtime,
                                                   ApplicationCoordinator* appCoord,
                                                   ILogicGateway* gateway)
{
    const ModuleUiAssemblyContext context{ mainWindow, runtime, appCoord, gateway, m_pageManager, m_globalUiManager };

    if (isModuleEnabled(QStringLiteral("review"))) {
        registerReviewModuleUi(context);
    }
}
```

### 6.8 第七步：把模块加到软件配置里

如果你希望它可见：

- `getEnabledModules()` 里加它
- `getModuleDisplayOrder()` 里加它
- 如果想开机默认进它，`getInitialModule()` 返回它

## 7. 新模块多个 UI 如何使用

这是用户问题里的重点之一。

在这个框架里，一个模块并不只有一个 UI。

一个模块可以同时拥有：

1. **一个主页面**
2. **多个右侧辅助 UI**
3. **多个底部辅助 UI**
4. **主页面内部多个 3D 窗口**

### 7.1 多个 UI 的核心机制

核心类是：

- `ModuleCoordinator`

它内部有：

- `setMainPage(QWidget* page)`：设置主页面
- `addAuxiliaryWidget(widget, Right)`：挂右侧辅助 UI
- `addAuxiliaryWidget(widget, Bottom)`：挂底部辅助 UI

### 7.2 真实例子 1：一个主页面 + 一个右侧面板

参考：

- `PointPickPage`
- `PointPickStatusPanel`

装配代码在：

- `src/app/software/ModuleUiAssemblers.cpp`

```cpp
auto* page = new PointPickPage();
auto* statusPanel = new PointPickStatusPanel(context.mainWindow->getWorkspaceShell());

coordinator->setMainPage(page);
coordinator->addAuxiliaryWidget(
    statusPanel,
    ModuleCoordinator::AuxiliaryRegion::Right);
```

### 7.3 真实例子 2：一个主页面内部两个 3D UI

参考：

- `src/modules/planning/PlanningPage.h`
- `src/app/software/ModuleUiAssemblers.cpp`

```cpp
auto* planningWindow = new VtkSceneWindow("planning_main", sceneGraph, page);
auto* overviewWindow = new VtkSceneWindow("planning_overview", sceneGraph, page);

page->setSceneWindow(planningWindow);
page->setSecondarySceneWindow(overviewWindow);
```

也就是说：

- 一个模块可以只有一个主页面
- 但主页面内部自己再摆多个功能区、多个 VTK 窗口、多个子控件

### 7.4 真实例子 3：多个右侧面板

当前仓库虽然没有一个模块一次性挂多个右侧面板，但 `ModuleCoordinator` 支持这么干。

你可以这样写：

```cpp
coordinator->addAuxiliaryWidget(summaryPanel, ModuleCoordinator::AuxiliaryRegion::Right);
coordinator->addAuxiliaryWidget(statusPanel, ModuleCoordinator::AuxiliaryRegion::Right);
coordinator->addAuxiliaryWidget(toolPanel, ModuleCoordinator::AuxiliaryRegion::Right);
```

它们会按添加顺序依次堆到右侧。

### 7.5 多个 UI 的推荐分工

很重要，建议这样设计：

#### 主页面放什么

- 主流程交互
- 主按钮
- 主 3D 视图
- 表单主体

#### 右侧面板放什么

- 状态摘要
- 当前步骤说明
- 小工具
- 只读信息

#### 底部面板放什么

- 短操作栏
- 状态条
- 小型控制条
- 进度 / 心跳 / 网络状态

## 8. 如何将模块的 UI 放在指定主窗口位置

这也是重点问题。

主窗口区域不是随便放的，项目已经固定好了挂载规则。

### 8.1 主窗口有哪些位置

看：

- `src/shell/WorkspaceShell.h`
- `src/shell/WorkspaceShell.cpp`

主要位置如下：

#### 位置 1：中间主区域

对应：

- `WorkspaceShell::getCenterStack()`

作用：

- 放模块主页面
- 一次只显示一个当前模块页面

装配方式：

```cpp
coordinator->setMainPage(page);
context.pageManager->registerPage("review", page);
```

#### 位置 2：右侧模块辅助区域

对应：

- `WorkspaceShell::mountRightAuxiliary(widget)`

但一般你不直接调它，而是：

```cpp
coordinator->addAuxiliaryWidget(widget, ModuleCoordinator::AuxiliaryRegion::Right);
```

模块激活时，`ApplicationCoordinator` 会自动把它挂到右侧。

#### 位置 3：底部模块辅助区域

对应：

- `WorkspaceShell::mountBottomAuxiliary(widget)`

一般也不手动调，而是：

```cpp
coordinator->addAuxiliaryWidget(widget, ModuleCoordinator::AuxiliaryRegion::Bottom);
```

#### 位置 4：右侧壳层区域

对应：

- `WorkspaceShell::getRightShellHost()`

这个位置通常给 **全局壳层模块** 用，不是给普通业务模块用。

示例：

- `ModuleNavigationModule`

在：

- `src/app/software/ConcreteSoftwareInitializers/DefaultSoftwareInitializer.cpp`

```cpp
if (QWidget* rightShellHost = workspaceShell->getRightShellHost()) {
    rightShellHost->layout()->addWidget(workflowMenu);
}
```

#### 位置 5：底部壳层区域

对应：

- `WorkspaceShell::getBottomShellHost()`

示例：

- `ModuleStatusBarModule`

```cpp
if (QWidget* bottomShellHost = workspaceShell->getBottomShellHost()) {
    bottomShellHost->layout()->addWidget(statusBar);
}
```

### 8.2 模块 UI 到底该放哪里

给你一个最实用判断表：

| 需求 | 应该放哪里 |
|---|---|
| 模块主要界面 | CenterStack |
| 模块自己的状态面板 | Right Auxiliary |
| 模块自己的小工具栏 | Bottom Auxiliary |
| 全局流程菜单 | Right Shell Host |
| 全局状态条 | Bottom Shell Host |

### 8.3 激活 / 切换时如何自动挂载

`ApplicationCoordinator::setCurrentModule()` 做了这件事。

它会：

1. 清掉旧模块右侧附属 UI
2. 清掉旧模块底部附属 UI
3. 切换中间页面
4. 把新模块的右侧附属 UI 挂上去
5. 把新模块的底部附属 UI 挂上去
6. 激活新模块 coordinator

也就是说：

> 你平时只负责 `addAuxiliaryWidget()`，不用自己写“切换时挪位置”的代码。

## 9. 如何切换模块

这也是高频需求。

### 9.1 模块切换完整链路

切模块不是 UI 直接改页面，而是走一条完整动作链。

#### 第一步：发切换请求

通常由壳层 UI 发出：

```cpp
appCoord->requestSwitchModule("planning");
```

在当前实现里，壳层 widget 也可以直接使用注入进去的 dispatcher：

```cpp
m_actionDispatcher->requestModuleSwitch("planning");
```

#### 第二步：`ApplicationCoordinator` 发壳层动作

```cpp
if (m_actionDispatcher) {
    m_actionDispatcher->requestModuleSwitch(moduleId);
}
```

#### 第三步：`LogicRuntime` 处理切换

在：

- `src/logic/runtime/LogicRuntime.cpp`

```cpp
switchToModule(targetModule, action.actionId);
```

#### 第四步：旧模块逻辑退场

```cpp
oldHandler->onModuleDeactivated();
```

#### 第五步：更新当前模块状态

```cpp
m_activeModuleState->setCurrentModule(targetModule);
```

#### 第六步：新模块逻辑进场

```cpp
newHandler->onModuleActivated();
```

#### 第七步：发出通知

`LogicRuntime` 会发：

- `ModuleChanged`
- `ActiveModuleChanged`

#### 第八步：UI 层切换页面和挂载辅助 UI

`ApplicationCoordinator::onShellNotification()` 收到后调用：

```cpp
setCurrentModule(moduleId);
```

它内部再做：

- `PageManager::switchToPage(moduleId)`
- 重新挂右侧/底部模块 UI

### 9.2 最常见切模块入口

#### 方式 1：壳层菜单切换

示例：

- `ModuleNavigationModule`

#### 方式 2：logic 收到服务器控制消息切换

在 `LogicRuntime::onServerCommandReceived()` 里支持：

- `switch_module`
- `request_switch_module`

#### 方式 3：初始启动自动切换

`BaseSoftwareInitializer::initialize()` 最后会主动发送：

```cpp
if (m_appCoordinator && m_appCoordinator->getActionDispatcher()) {
    m_appCoordinator->getActionDispatcher()->requestModuleSwitch(initialModule);
}
```

### 9.3 如果你要在模块内部切到别的模块

比如当前模块做完后，自动跳下一模块，你可以从 logic 发通知给 UI 或直接让 UI 发壳层动作。

最稳妥做法是：

- logic 发一个状态通知
- 页面或壳层收到后调用 `ApplicationCoordinator::requestSwitchModule()`

如果你一定要纯 logic 发起，也可以通过控制消息路径或者扩展 gateway。

## 10. 如何使用模块 A 的 logic 控制模块 A 的 UI

这个问题非常重要，答案是：

> **通过 `LogicNotification`，不要让 logic 直接拿着 UI 指针改界面。**

### 10.1 正确思路

logic 只负责发“状态变化通知”。

UI 通过 `ModuleCoordinator::notificationForPage` 接通知，然后自己刷新控件。

### 10.2 真实示例：PointPick

logic 在：

- `src/modules/pointpick/PointPickModuleLogicHandler.cpp`

它在 `emitSelectionState()` 里发通知：

```cpp
LogicNotification notification = LogicNotification::create(
    LogicNotification::SceneNodesUpdated,
    LogicNotification::CurrentModule,
    payload);
emit logicNotification(notification);
```

payload 里带了：

- `points`
- `pointCount`
- `confirmed`

UI 装配时接这个通知：

```cpp
QObject::connect(coordinator, &ModuleCoordinator::notificationForPage,
                 page, [page, statusPanel](const LogicNotification& notification) {
                     if (notification.eventType != LogicNotification::SceneNodesUpdated) {
                         return;
                     }

                     page->updatePointList(notification.payload.value("points").toStringList());
                     page->setPointCount(notification.payload.value("pointCount").toInt());
                     page->setConfirmed(notification.payload.value("confirmed").toBool());

                     statusPanel->setPointCount(notification.payload.value("pointCount").toInt());
                     statusPanel->setConfirmed(notification.payload.value("confirmed").toBool());
                 });
```

### 10.3 你应该怎么写

#### logic 侧

```cpp
QVariantMap payload;
payload.insert("status", "approved");
payload.insert("progress", 100);

emit logicNotification(LogicNotification::create(
    LogicNotification::CustomEvent,
    LogicNotification::CurrentModule,
    payload));
```

#### UI 装配侧

```cpp
QObject::connect(coordinator, &ModuleCoordinator::notificationForPage,
                 page, [page](const LogicNotification& notification) {
                     if (notification.eventType != LogicNotification::CustomEvent) {
                         return;
                     }
                     page->setStatusText(notification.payload.value("status").toString());
                 });
```

### 10.4 小白牢记一句话

> **logic 发数据，UI 自己决定怎么显示。**

不要在 logic 里：

- include 某个 Page 头文件
- 保存 Page 指针
- 直接调用 `page->setText(...)`

这会把模块耦死。

## 11. 如何使用模块 A logic 控制模块 B logic

这个问题在本项目里已经有标准方案：

> **用 `invokeModule()`**

### 11.1 核心机制

`ModuleLogicHandler` 提供：

```cpp
ModuleInvokeResult invokeModule(const QString& targetModule,
                                const QString& method,
                                const QVariantMap& payload = {});
```

底层会走到：

- `LogicRuntime::invokeModule(...)`
- 找到目标模块 handler
- 调目标模块的 `handleModuleInvoke(...)`

### 11.2 目标模块要先支持被调用

比如 `datagen` 就支持：

- `src/modules/datagen/DataGenModuleLogicHandler.cpp`

它实现了：

```cpp
ModuleInvokeResult DataGenModuleLogicHandler::handleModuleInvoke(const ModuleInvokeRequest& request)
```

所以别的模块就可以调它。

### 11.3 推荐用法

假设模块 A 想让模块 B 创建一个点节点：

```cpp
const ModuleInvokeResult result = invokeModule(
    QStringLiteral("datagen"),
    QStringLiteral("create_node"),
    {
        {QStringLiteral("nodeType"), QStringLiteral("point")},
        {QStringLiteral("name"), QStringLiteral("Relay Points")},
        {QStringLiteral("count"), 4}
    });

if (!result.ok) {
    emitInvokeFailureNotification(result, QStringLiteral("datagen"));
    return;
}
```

### 11.4 什么时候用 invokeModule

适合：

- A logic 需要调用 B logic 的业务能力
- A 不关心 B 的 UI
- 想保留模块边界

不适合：

- 只是想切 UI 页签
- 只是想更新一个全局状态栏

### 11.5 现有项目中的典型场景

当前仓库的 UI 层已经有“跨模块命令”示例：

- `params` 页面可以触发 `datagen`
- `pointpick` 页面可以触发 `datagen`
- `planning` 页面可以触发 `datagen`
- `navigation` 页面可以触发 `datagen`

这些是通过 Action 路由到目标模块实现的。

如果你想在 **logic 内部** 做同样事情，建议直接用 `invokeModule()`，更清晰。

## 12. 如何使用模块 A logic 控制模块 B 的 UI

这个问题很多人一开始会理解成：

> A logic 能不能直接拿到 B 的页面，然后改 B 页面？

**不建议这样做。**

正确做法有两种。

### 12.1 推荐方式 1：A logic -> B logic -> B UI

也就是：

1. A logic 调用 B logic
2. B logic 根据业务规则更新自己的状态
3. B logic 发自己的 `LogicNotification`
4. B UI 接通知更新

这是最标准、最干净的方式。

### 12.2 推荐方式 2：A logic 修改共享 SceneGraph / 共享状态，B UI 自然变化

这是当前项目最常见的跨模块影响 UI 方式。

例如：

- A 模块创建或修改节点
- B 模块的 3D 页面本来就在显示同一个 `SceneGraph`
- 所以 B UI 会自动看到变化

### 12.3 真实案例：A 影响 B UI 的最实际示例

#### 场景 1：通过共享 SceneGraph 影响

比如 `pointpick` / `planning` / `navigation` 都能触发 `datagen` 创建节点。

`datagen` 创建节点后：

- 节点进入统一 `SceneGraph`
- 各模块 `VtkSceneWindow` 都在监听同一个场景
- 所以别的模块 3D UI 会显示新节点

这本质上就是：

> 模块 A 影响了模块 B 的 UI，只是中间通过共享场景完成，而不是直接拿指针。

#### 场景 2：通过调用目标模块 logic

比如 A logic 调 B logic：

```cpp
invokeModule("planning", "set_status", {
    {"status", "review_finished"}
});
```

然后你可以在 `PlanningModuleLogicHandler` 里补充 `handleModuleInvoke()`，更新状态后再发通知给 `PlanningPage`。

### 12.4 最推荐的设计原则

如果你想让 A 改 B UI，优先顺序是：

1. **共享 SceneGraph**
2. **A logic 调 B logic**
3. **公共全局状态 + B 自己刷新**
4. **最后才考虑直接拿 UI 指针（基本不推荐）**

### 12.5 千万别这么干

不要在 A logic 里写：

```cpp
PlanningPage* page = ...;
page->setPlanStatus("xxx");
```

原因：

- 强耦合
- 生命周期难管
- 容易崩
- 模块切换后容易出现野指针/空指针

## 14. 如何使用模块的 UI 控制当前模块的 logic

这是最基础的日常写法。

### 14.1 标准流程

1. `ModuleUiAssemblers.cpp` 给页面注入 `coordinator->getActionDispatcher()`
2. 页面按钮点击后自己构建 `command` / `payload`
3. 页面调用 `UiActionDispatcher::sendCommand()` 或 `sendTargetedCommand()`
4. `ILogicGateway` 把动作送进 `LogicRuntime`
5. 当前模块 `handleAction()` 解析 `payload.command`

补充说明：

- `ModuleCoordinator::sendModuleAction()` 仍然存在，主要用于兼容旧代码或少量非页面场景
- 新模块默认优先使用 `UiActionDispatcher`

### 14.2 真实示例：PointPick 页面确认按钮

页面里：

```cpp
connect(m_ui->confirmButton, &QPushButton::clicked,
        this, [this]() {
            if (m_actionDispatcher) {
                m_actionDispatcher->sendCommand(PointPickUiCommands::confirmPoints());
            }
        });
```

logic 里：

```cpp
const QString command = action.payload.value(QStringLiteral("command")).toString().trimmed();

if (action.actionType == UiAction::ConfirmPoints ||
    command == PointPickUiCommands::confirmPoints()) {
    m_confirmed = true;
    emitSelectionState(action.actionId);
    return;
}
```

### 14.3 表单类页面示例：Params

页面里：

```cpp
connect(m_applyButton, &QPushButton::clicked, this, [this]() {
    if (!m_actionDispatcher) {
        return;
    }

    m_actionDispatcher->sendCommand(
        ParamsUiCommands::applyParameters(),
        {{QStringLiteral("parameters"), collectParameters()}});
});
```

logic 里：

```cpp
if (action.actionType == UiAction::CustomAction) {
    const QString command = action.payload.value(QStringLiteral("command")).toString().trimmed();
    if (command != ParamsUiCommands::applyParameters()) {
        return;
    }

    const QVariantMap parameters = action.payload.value(QStringLiteral("parameters")).toMap();
    for (auto it = parameters.cbegin(); it != parameters.cend(); ++it) {
        m_parameters.insert(it.key(), it.value());
    }
}
```

注意这里和旧版本最大的不同是：

> `ParamsPage` 现在一次点击会把整张表单打成一个 `parameters` map 发出去，而不是把每个字段拆成一条 `UpdateParameter` action。这样更省装配代码，也更符合“页面自己构建消息”的设计。

### 14.4 如果页面想控制“当前激活模块”而不是固定模块

通常不用你额外处理。

因为 `ModuleCoordinator` 本身就是绑到当前模块 ID 的。

只要你在模块装配里写的是：

```cpp
auto* coordinator = new ModuleCoordinator("review", ...);
```

那页面动作自然就会发到 `review` 模块 logic。

## 16. 如何在轮询的 Redis 中控制模块的 logic

你的问题可以理解成：

> Redis 里某个 key 周期性被读取后，怎样让这个数据驱动模块 logic？

答案是：

> **通过 `PollingTask -> CommunicationHub -> LogicRuntime::onStateSampleReceived -> handleStateSample()`**

### 16.1 轮询入口在哪里配置

在软件初始化器里配置。

参考：

- `src/app/software/ConcreteSoftwareInitializers/DefaultSoftwareInitializer.cpp`

当前默认做法是：每个启用模块都自动注册一个 polling key。

```cpp
const QString pollingKey = pollingKeyForModule(profile, moduleId);
auto* task = new PollingTask(
    QStringLiteral("%1_poll").arg(moduleId),
    moduleId,
    pollingKey,
    pollingIntervalForModule(profile, moduleId));
task->setLatestWins(true);
task->setChangeDetection(true);
task->setMaxDispatchRateHz(dispatchRateForModule(profile, moduleId));
commHub->addPollingTask(task);
```

### 16.2 轮询数据怎么进入你的模块

关键点是 `PollingTask` 构造时带了 `moduleId`。

比如：

```cpp
new PollingTask("review_poll", "review", "state.review.latest", 100);
```

那这条数据最终就会路由到：

```cpp
ReviewModuleLogicHandler::handleStateSample(const StateSample& sample)
```

### 16.3 你的 logic 应该怎么写

```cpp
void ReviewModuleLogicHandler::handleStateSample(const StateSample& sample)
{
    QVariantMap payload = sample.data.value(QStringLiteral("value")).toMap();
    if (payload.isEmpty()) {
        payload = sample.data;
    }

    const QString status = payload.value(QStringLiteral("status")).toString();
    if (!status.isEmpty()) {
        m_status = status;
    }

    emit logicNotification(LogicNotification::create(
        LogicNotification::CustomEvent,
        LogicNotification::CurrentModule,
        {{"status", m_status}, {"sourceSampleId", sample.sampleId}}));
}
```

### 16.4 真实示例 1：PointPick 轮询点集

`PointPickModuleLogicHandler::handleStateSample()` 会：

- 解析 points
- 更新 `PointNode`
- 更新 confirmed
- 发通知刷新页面

### 16.5 真实示例 2：Navigation 高频轮询变换

`NavigationModuleLogicHandler::handleStateSample()` 会：

- 识别 `matrixToParent`
- 更新 TransformNode
- 更新导航状态
- 发通知给 `NavigationPage`

### 16.6 轮询适合什么数据

适合：

- 最新状态快照
- 定位/位姿
- 参数镜像
- 监控值

不太适合：

- 严格不能丢的事件流
- 一次性命令

## 17. 如何在 Redis 订阅中调用 logic

这个问题可以理解成：

> Redis PubSub 某个频道来了消息，怎么让模块 logic 响应？

答案是：

> **通过 `SubscriptionSource -> CommunicationHub -> LogicRuntime::onStateSampleReceived -> handleStateSample()`**

### 17.1 订阅入口在哪配置

还是在软件初始化器里。

参考：

- `src/app/software/ConcreteSoftwareInitializers/DefaultSoftwareInitializer.cpp`

```cpp
const QString channel = subscriptionChannelForModule(profile, moduleId);
if (!channel.isEmpty()) {
    commHub->addSubscriptionSource(new SubscriptionSource(
        QStringLiteral("%1_subscription").arg(moduleId),
        channel,
        moduleId));
}
```

### 17.2 订阅消息最终进哪里

最终和轮询一样，也进：

```cpp
handleStateSample(const StateSample& sample)
```

这点很关键：

> **轮询数据和订阅数据，在模块侧的处理入口是统一的。**

这意味着你写模块时很省心：

- 不需要为“轮询”和“订阅”写两套处理函数
- 一般只需要统一做 payload 归一化

### 17.3 推荐写法

```cpp
QVariantMap normalizedSampleData(const StateSample& sample)
{
    QVariantMap payload = sample.data.value(QStringLiteral("value")).toMap();
    return payload.isEmpty() ? sample.data : payload;
}
```

这个写法在多个模块里都出现了，比如：

- `PointPickModuleLogicHandler.cpp`
- `PlanningModuleLogicHandler.cpp`
- `NavigationModuleLogicHandler.cpp`

### 17.4 什么时候该用订阅，不该用轮询

#### 用订阅

- 状态变化不固定频率
- 希望收到就立刻响应
- 服务器主动推送

#### 用轮询

- 想持续拉最新状态
- 允许只取最新值
- 采样型数据

## 18. 如何在 Redis 控制消息中调用 logic

上面讲的是“状态流”。

这里再讲一个更直接的控制路径：

> **Redis 控制消息 -> 直接转成 UiAction -> 调 logic `handleAction()`**

### 18.1 入口在哪

`BaseSoftwareInitializer::initialize()` 中会连接：

```cpp
QObject::connect(commHub, &CommunicationHub::controlMessageReceived,
                 logicRuntime, &LogicRuntime::onControlMessageReceived);
```

### 18.2 `LogicRuntime::onControlMessageReceived()` 做了什么

它会：

1. 校验消息 seq，避免旧消息覆盖新消息
2. 解析 `actionType`
3. 推断目标模块
4. 构造一个 `UiAction`
5. 直接调用 `onActionReceived(action)`

也就是说：

> 服务器发来的控制消息，最终和“用户点按钮”走的是同一套 logic 路径。

### 18.3 控制消息示例

例如 Redis 控制面发来：

```json
{
  "actionType": "GeneratePlan",
  "module": "planning",
  "payload": {
    "status": "planning_complete"
  },
  "seq": 1001
}
```

那么最终会进入：

```cpp
PlanningModuleLogicHandler::handleAction(...)
```

### 18.4 服务端切模块示例

`LogicRuntime::onServerCommandReceived()` 还支持：

- `switch_module`
- `request_switch_module`
- `next_step`
- `prev_step`
- `resync`

比如：

```json
{
  "commandType": "switch_module",
  "payload": {
    "targetModule": "navigation"
  },
  "msgId": "server-001"
}
```

最终就会切到 `navigation` 模块。

### 19.4 第三步：写 logic

```cpp
#include "ReviewUiCommands.h"

class ReviewModuleLogicHandler : public ModuleLogicHandler
{
    Q_OBJECT

public:
    explicit ReviewModuleLogicHandler(QObject* parent = nullptr)
        : ModuleLogicHandler(QStringLiteral("review"), parent)
    {
    }

    void handleAction(const UiAction& action) override
    {
        if (action.actionType != UiAction::CustomAction) {
            return;
        }

        const QString command = action.payload.value(QStringLiteral("command")).toString().trimmed();

        if (command == ReviewUiCommands::approve()) {
            m_status = QStringLiteral("approved");
            writeRedisJsonValue(QStringLiteral("state.review.latest"),
                                {{"status", m_status}});

            emit logicNotification(LogicNotification::create(
                LogicNotification::CustomEvent,
                LogicNotification::CurrentModule,
                {{"status", m_status}}));
            return;
        }

        if (command == ReviewUiCommands::createDemoNode()) {
            const ModuleInvokeResult result = invokeModule(
                QStringLiteral("datagen"),
                QStringLiteral("create_node"),
                {
                    {QStringLiteral("nodeType"), QStringLiteral("point")},
                    {QStringLiteral("name"), QStringLiteral("Review Demo Points")},
                    {QStringLiteral("count"), 3}
                });

            if (!result.ok) {
                emitInvokeFailureNotification(result, QStringLiteral("datagen"), action.actionId);
            }
        }
    }

    void handleStateSample(const StateSample& sample) override
    {
        QVariantMap payload = sample.data.value(QStringLiteral("value")).toMap();
        if (payload.isEmpty()) {
            payload = sample.data;
        }

        const QString status = payload.value(QStringLiteral("status")).toString();
        if (!status.isEmpty()) {
            m_status = status;
        }

        emit logicNotification(LogicNotification::create(
            LogicNotification::CustomEvent,
            LogicNotification::CurrentModule,
            {{"status", m_status}, {"sourceSampleId", sample.sampleId}}));
    }

    void onModuleActivated() override
    {
        if (hasRedisCommandAccess()) {
            const QVariantMap snapshot = readRedisJsonValue(QStringLiteral("state.review.latest"));
            const QString status = snapshot.value(QStringLiteral("status")).toString();
            if (!status.isEmpty()) {
                m_status = status;
            }
        }

        emit logicNotification(LogicNotification::create(
            LogicNotification::CustomEvent,
            LogicNotification::CurrentModule,
            {{"status", m_status.isEmpty() ? QStringLiteral("idle") : m_status}}));
    }

private:
    QString m_status;
};
```

void registerReviewModuleUi(const ModuleUiAssemblyContext& context)
{
    auto* coordinator = new ModuleCoordinator(

}

## 20. 多个模块之间到底应该怎样协作

这一节把上面内容再压缩成“开发规则”。

### 20.1 模块之间最推荐的三种协作方式

#### 方式 1：调用目标模块 logic

适用：

- 需要目标模块提供业务能力

工具：

- `invokeModule()`

#### 方式 2：共享 SceneGraph

适用：

- 需要多个模块共享 3D 数据

工具：

- `getSceneGraph()`
- PointNode / LineNode / ModelNode / TransformNode

#### 方式 3：通过 `LogicNotification` 回推 UI

适用：

- logic 状态改变后通知页面刷新

工具：

- `emit logicNotification(...)`

### 20.2 不推荐的协作方式

#### 不推荐 1：logic 直接拿别的模块页面指针

问题：

- 耦合严重
- 生命周期危险

#### 不推荐 2：页面之间互相直接调

问题：

- 页面变复杂
- 逻辑散在 UI 层

#### 不推荐 3：页面直接读写 Redis

问题：

- UI 和通信耦死
- 以后难测、难改、难维护

## 24. 最后给你的开发建议

如果你真的是第一次基于这个框架写完整软件，我建议你按下面顺序做：

### 第 1 步

先复制一个最简单模块：

- 推荐从 `pointpick` 开始复制

因为它已经覆盖了：

- 页面
- 侧栏
- logic
- 通知
- 节点
- 轮询

### 第 2 步

先让模块跑通 **UI -> logic -> UI** 这条链。

也就是先做到：

- 点按钮
- logic 收到
- 页面状态变

### 第 3 步

再接 `SceneGraph`。

先创建一个最简单 `PointNode`，确保能在 VTK 窗口显示。

### 第 4 步

再接 Redis。

先做：

- `state.xxx.latest` 轮询

跑通以后再做：

- `state.xxx` 订阅
- `control.downstream` 控制消息

### 第 5 步

最后再做模块间联动。

优先顺序还是那句话：

1. 共享 SceneGraph
2. `invokeModule()`
3. `LogicNotification`

不要一上来就跨模块互拿 UI 指针。

## 25. 一句话总结

如果你只记住一句话，请记住这句：

> **这个框架的正确写法是：UI 自己构建动作并通过 `UiActionDispatcher` 发出去，logic 只管业务和状态，模块之间通过 invokeModule / SceneGraph / Notification 协作，Redis 只进 logic，不直接进 UI。**

按这个原则写，你的软件就会越写越稳，越写越容易扩展。
